# Инструкция конструктора runtime-symbols шаблонов.

1. Будет питоновский конструктор (`RUNTIME_FSM/runtime_builder.py`), который будет брать список символов заданных в CFG/app.json -> "symbols".
2. Опираясь на ЧИСТЫЙ шаблон пользовательских настроек `CFG/_base.json`, фабрика создает пер-символьный кеш типа btcusdt.json (добавляя туда все нужные рантайм-флаги типа `total_volume`, `is_active`, `in_position`), который будет лежать по пути `CFG/runtime/`.
3. Его структура синхронизирована со структурой PositionState (включая все новые поля и состояния) принцип -- одно является зеркальной копией другого.
4. Это правило применяется к каждому символу в списке symbols при запуске (если файл не сгенерирован).
5. Остальная часть управления и редактирования состояний этих файлов прописана в других инструкциях кода.
6. При старте бота через Telegram (кнопка `Start`), бот выводит баннер-сводку текущих конфигураций (вкл/выкл, объем) и просит подтвердить `Confirm Start` или перенастроить. Эта опция страхует от случайного запуска со старыми настройками.
7. Данные раскиданы по папкам для порядка: `CFG/` (только конфиги), `ANALYTICS/` (csv и json аналитика), `CACHE/` (системная память, например кэш выставленных плечей).

### Уточнения по структуре кеша рантайма (FSM -> JSON):
- `_base.json` содержит ТОЛЬКО голые настройки (`enable`, `invest_size`, `leverage`, `grid`, `tp_map` без цен и `is_active`).
- Удалено старое поле `tp_id`. ID ордеров больше не кэшируются локально (вместо этого бот парсит актуальные ордера с биржи).
- Поля `price` и `is_active` добавляются на лету фабрикой при генерации файлов `runtime/` в уровни сетки (`grid`) и в карту тейк-профитов (`tp_map`). 
- Вся готовая структура в `CFG/runtime/` (включая динамические ключи со значениями `null` / `false`) строго зеркалирует то, чем оперирует `PositionState` во время торговли.
